# Вкладка Аудит

Вкладка Аудит используется ролью **Аудитор / Исследователь** для
проверки целостности выборов через анализ on-chain событий.

---

## Раздел 1 — Результаты

Показывает текущие подсчёты голосов. Обновляется когда:

- Запущен аудит.
- Пользователь нажал **Обновить** вручную.

| Элемент | Описание |
|---|---|
| **Победитель** | Имя кандидата и партия. Показывает "Ничья: A, B" при равенстве голосов, "Победитель не определён" при отсутствии голосов. Отображается только после стадии `FINISHED`. |
| **Таблица кандидатов** | Колонки: Кандидат / Партия / Адрес / Голоса. Сортировка по голосам по убыванию. |
| **Обновить** | Перезагружает результаты из цепи без запуска проверок. |

---

## Раздел 2 — Проверки безопасности

Основной интерфейс аудита.

### Селектор режима аудита

Четыре режима, динамически включающиеся / выключающиеся в зависимости
от стадии контракта:

| Режим | Активен в стадии | Что выполняет |
|---|---|---|
| **Полный аудит** | FINISHED | Все проверки (pre + live + final). |
| **Предголосовая проверка** | SETUP | Количество кандидатов, заполненность whitelist, валидность стадии. |
| **Текущая проверка** | ACTIVE | Двойное голосование, контроль whitelist. |
| **Финальный аудит** | FINISHED | Валидация кандидатов, контроль стадий, целостность подсчёта, действия владельца. |

Недоступные режимы показывают суффикс "(недоступен)" в комбобоксе,
а кнопка **Запустить аудит** заблокирована.

### Запустить аудит

Запускает выбранный режим аудита. Результаты появляются в таблице
проверок:

| Колонка | Описание |
|---|---|
| **Проверка** | Название (например, "Double Vote Protection"). |
| **Статус** | Цветной бейдж: `PASSED` (зелёный), `FAILED` (красный), `WARNING` (жёлтый), `SKIPPED` (серый). |
| **Детали** | Понятное объяснение с доказательством (например, "No duplicate voters", "Events=87 OnChain=87"). |

### Строка сводки

Под таблицей: `Аудит YYYY-MM-DD HH:MM:SS — Пройдено: N, Не пройдено: K, Всего: T`.

## Раздел 3 — Экспорт

Три кнопки:

| Кнопка | Результат |
|---|---|
| **Копировать отчёт** | Копирует полный отчёт в буфер обмена как форматированный JSON (результаты + проверки + победитель). |
| **JSON** | Сохраняет полный отчёт в файл `.json`. |
| **CSV** | Сохраняет отчёт в `.csv` (секции: Results, Winner, Audit). |

Все три используют **единый источник истины** — `AppController.build_full_report()`.

## Справочник SEC-проверок

Детальная спецификация всех 6 инвариантов:
[Проверки безопасности SEC-01..06](../security/sec-checks.ru.md).

Краткая сводка:

| Код | Проверка | Защита в контракте | Верификация аудитом |
|---|---|---|---|
| SEC-01 | Защита от двойного голосования | `hasVoted[msg.sender]` | Нет дубликатов `VoteCast.voter` |
| SEC-02 | Контроль whitelist | `require(whitelist[msg.sender])` | Каждый `VoteCast.voter` ∈ `VoterWhitelisted` |
| SEC-03 | Контроль стадии | `onlyStage(Stage.Active)` | Все `VoteCast.blockNumber` ∈ [start, end] |
| SEC-04 | Валидация кандидата | `require(candidates[c].registered)` | Каждый `VoteCast.candidate` ∈ `CandidateAdded` |
| SEC-05 | Owner-only администрирование | модификатор `onlyOwner` | Admin TX `from` == `owner()` контракта |
| SEC-06 | Целостность подсчёта голосов | неявно | `len(VoteCast events) == sum(candidate.vote_count)` |

## Типичный сценарий

1. После стадии `FINISHED` (администратор нажал Завершить голосование).
2. Откройте вкладку Аудит.
3. Выберите **Полный аудит**.
4. Нажмите **Запустить аудит**.
5. Просмотрите статусы проверок.
6. Нажмите **JSON** для сохранения отчёта в архив.
