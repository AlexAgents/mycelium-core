# Спецификация инвариантов безопасности (SEC-01..06)

## Заметка аналитика: Двухуровневая архитектура защиты
В MYCELIUM CORE инварианты безопасности обеспечиваются с использованием модели **Двухуровневой защиты (Dual-Layer Defense)**:

1. **Проактивная защита On-Chain:** Смарт-контракт `VotingCore.sol` активно отклоняет несанкционированные транзакции с помощью конструкций Solidity `require()` и кастомных ошибок.
2. **Реактивный Off-Chain Аудит:** Python-сервис `AuditService` независимо реконструирует состояние, считывая неизменяемые журналы событий (`VoteCast`, `StageChanged`), математически проверяя, что контракт вел себя именно так, как ожидалось.

Этот дуализм гарантирует, что даже если в управлении состоянием контракта появится логическая уязвимость, независимый аудит на основе событий выявит расхождение.

## SEC-01: Защита от повторного голосования

- **Правило:** Адрес избирателя не может проголосовать более одного раза.
- **Проактивно (Контракт):** `hasVoted[msg.sender] = true` при выполнении; `revert AlreadyVoted()` при повторной попытке.
- **Реактивно (Аудит):** Подсчитывает события `VoteCast` для каждого адреса избирателя. Любой адрес со счётчиком > 1 переводит проверку в `FAILED`.
- **Код:** `AuditService._check_double_vote()`

## SEC-02: Соблюдение ограничений whitelist

- **Правило:** Голосовать могут только адреса из whitelist.
- **Проактивно (Контракт):** `if (!whitelist[msg.sender]) revert NotWhitelisted()`.
- **Реактивно (Аудит):** Для каждого события `VoteCast` проверяет наличие избирателя в whitelist через `VotingService.is_whitelisted(voter)`. Любой адрес не из whitelist переводит проверку в `FAILED`.
- **Код:** `AuditService._check_whitelist_enforcement()`

## SEC-03: Соблюдение стадий голосования

- **Правило:** Голоса могут приниматься только на стадии `Active`.
- **Проактивно (Контракт):** Модификатор `onlyStage(Stage.Active)` на функции `castVote()`.
- **Реактивно (Аудит):** Считывает номера блоков из событий `VotingStarted` и `VotingFinished`. Проверяет, что номера блоков всех событий `VoteCast` попадают в диапазон `[start_block, end_block]`.
- **Запасной вариант:** При недоступности событий стадий проверяет, что текущее название стадии — `ACTIVE` или `FINISHED` (со статусом `WARNING`).
- **Код:** `AuditService._check_stage_enforcement()`

## SEC-04: Валидация кандидатов

- **Правило:** Голоса могут подаваться только за зарегистрированных кандидатов.
- **Проактивно (Контракт):** `if (!candidates[candidate].registered) revert CandidateNotFound()`.
- **Реактивно (Аудит):** Формирует множество адресов зарегистрированных кандидатов через `get_candidates()`. Для каждого события `VoteCast` проверяет наличие `event.candidate` в этом множестве.
- **Код:** `AuditService._check_candidate_validation()`

## SEC-05: Администрирование только владельцем

- **Правило:** Все административные действия (смена стадий, добавление кандидатов) должны выполняться владельцем контракта.
- **Проактивно (Контракт):** Модификатор `onlyOwner` на функциях `addCandidate()`, `addVotersBatch()`, `startVoting()`, `finishVoting()`.
- **Реактивно (Аудит):** Считывает события `CandidateAdded`, `VoterWhitelisted` и `StageChanged`. Для каждого события сравнивает отправителя (`tx.from`) с `contract.owner()`. Любое несоответствие переводит проверку в `FAILED`.
- **Код:** `AuditService._check_owner_actions()`

## SEC-06: Целостность подсчёта голосов

- **Правило:** Общее количество событий `VoteCast` должно совпадать с суммой значений `vote_count` всех кандидатов, хранящихся в контракте.
- **Проактивно (Контракт):** Неявное — `candidates[candidate].votes += 1` при каждом вызове `castVote()`.
- **Реактивно (Аудит):** Независимо подсчитывает события `VoteCast`. Суммирует `vote_count` по всем кандидатам через `get_candidates()`. Сравнивает два числа. Любое расхождение означает состояние `FAILED`.
- **Код:** `AuditService._check_vote_count_integrity()`
