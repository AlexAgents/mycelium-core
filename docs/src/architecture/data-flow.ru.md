# Сценарии потоков данных

Этот документ описывает, как данные транзакций и состояний циркулируют между слоями системы при выполнении ключевых операций.

---

## 1. Прохождение голоса (Конвейер потока данных)

Когда избиратель отправляет голос, данные проходят через слои, изменяя состояние блокчейна, и возвращают пользователю криптографическую квитанцию.

```text
  [ Интерфейс Избирателя ]
        │  1. Ввод приватного ключа и адреса кандидата
        ▼
  [ AppController ]
        │  2. Делегирование precheck_vote()
        ├──────────────────────────┐
        ▼                          ▼
  [ Сервис Precheck ]       [ ErrorParser ]
        │  3. Валидно? (Да)        │ (В случае сетевой ошибки RPC)
        ▼                          ▼
  [ VotingService ] <──────────────┘
        │  4. Оценка газа, получение nonce, подпись транзакции
        ▼
  [ Web3Provider ]
        │  5. Отправка подписанной транзакции (JSON-RPC)
        ▼
  [ Локальный узел Geth ] ──(Изменение состояния)──> [ Контракт VotingCore ]
        │
        │  6. Возврат квитанции транзакции (Номер блока, затраченный газ)
        ▼
  [ Web3Provider ]
        │  7. Упаковка в DTO-модель VoteReceipt
        ▼
  [ Слой Представления (UI) ] ──> Генерация строки данных для QR-кода
```

## 2. Ключевые этапы

1. **Локальная проверка (Precheck):** Предотвращает отправку транзакций, которые заведомо завершатся ошибкой (например, избиратель уже проголосовал или отсутствует в whitelist). Это экономит ресурсы и предотвращает загрязнение сети.
2. **Синхронизация Nonce:** Контролируется `NonceManager` для предотвращения коллизий транзакций при последовательном выполнении нескольких вызовов.
3. **Генерация квитанции:** Возвращённая квитанция форматируется в строго типизированный объект `VoteReceipt` и передаётся обратно в GUI для отрисовки QR-кода.
